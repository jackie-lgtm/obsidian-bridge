# The Referral System

This doc explains how obsidian-bridge acts as a **conversational memory referral system** — surfacing past context during live conversation without burning tokens.

## Three data structures

### 1. Entity Index (`~/.claude/obsidian-bridge.entities.json`)

A flat, cheap JSON file. The heart of token-efficient retrieval.

```json
{
  "version": 1,
  "rebuilt_at": "2026-04-22T15:00",
  "entities": {
    "Project Alpha": {
      "type": "project",
      "aliases": ["alpha", "proj alpha", "the alpha project"],
      "memory_refs": ["mem_a7f3", "mem_b12e", "mem_c890"],
      "vault_notes": [
        "99 - Inbox/2026-04-22-1545 - t7a2 - customer-churn-observation.md",
        "10 - Projects/Project Alpha.md"
      ],
      "last_seen": "2026-04-22T15:45",
      "mention_count": 14
    },
    "Sarah": {
      "type": "person",
      "aliases": ["sarah m", "s."],
      "memory_refs": ["mem_d334", "mem_e991"],
      "vault_notes": ["20 - People/Sarah.md"],
      "last_seen": "2026-04-20T09:12",
      "mention_count": 8
    }
  }
}
```

**Key properties:**
- Target size: ≤ 10KB. At ~80 bytes per entity, that's ~125 entities.
- Sorted by `mention_count` descending so most-used entities are first in the file.
- **No full content** — just names, aliases, and refs. Content is pulled on demand.
- **Rebuilt weekly** (or on demand) by `scripts/build-entity-index.sh`.

### 2. claude-mem (canonical memory)

Stores full observations, session transcripts, vector embeddings. Queried via `mcp__plugin_claude-mem_mcp-search__*` tools. This is the **authoritative** store — if the entity index and claude-mem disagree, claude-mem wins.

### 3. Obsidian vault (human interface)

The human-readable layer. Every note has `memory-ref` frontmatter pointing back to a claude-mem observation ID.

## Retrieval tiers (cheap → expensive)

| Tier | Operation | Cost | When to use |
|---|---|---|---|
| 0 | Entity index lookup (string match) | ~0 tokens | Every turn |
| 1 | 1-line summary from index (`last_seen`, `mention_count`) | ~40 tokens | When entity detected |
| 2 | claude-mem summary (`mcp__...search` with `limit: 1`) | ~100-200 tokens | Brian asks "what about X?" |
| 3 | claude-mem timeline (context around anchor) | ~300-500 tokens | Brian asks "refresh my memory on X" |
| 4 | Full observation content (`get_observations`) | ~500-1500 tokens | Brian asks for detail |
| 5 | Full Obsidian note (Read tool) | variable | Brian asks for the note itself |

**Rule:** Always start at Tier 0-1. Only escalate when Brian engages.

## Entity detection algorithm

On every user turn:

1. Get Brian's input text.
2. Lowercase it.
3. For each entity in the index (sorted by `mention_count` desc for early-hit optimization):
   - Check if `entity.name` or any `alias` appears as a word in the input
   - Use word-boundary matching to avoid false positives (`alpha` shouldn't match `alphabet`)
4. Collect all hits. Deduplicate.
5. **Rank hits** by recency × mention_count — surface the top 1-2, not all of them.

**No LLM call.** This is pure string matching against a ≤10KB file. Effectively free.

## Surfacing protocol

When 1+ entities are detected in Brian's turn:

1. Answer his actual question first (his turn is the priority).
2. At the end of the response, append a **single-line referral**:
   ```
   ↪ *You mentioned **Project Alpha** — 14 past mentions, last context: Apr 18 (auth decision). Pull it up?*
   ```
3. If 2+ entities, combine them:
   ```
   ↪ *You mentioned **Project Alpha** and **Sarah** — both came up in the Apr 18 session. Pull that up?*
   ```
4. If 3+ entities, show the top one by recency only. Don't spam.

**Do NOT surface** if:
- Brian has already asked about the entity in this session (avoid repeating)
- The entity is trivial (generic words that matched by coincidence)
- Brian's current turn is a direct command (e.g., "fix this bug") — he doesn't need a memory referral while executing

## Clustering algorithm

Clusters are surfaced groupings of related thoughts. Generated by `scripts/build-entity-index.sh` and written to `{vault}/00 - Clusters/`.

**How clusters form:**

1. **Co-occurrence graph** — build a graph where nodes are entities and edges are weighted by how often they appear in the same claude-mem observation or the same Obsidian note.
2. **Community detection** — run a simple label-propagation or modularity algorithm to find tightly-connected clusters.
3. **Minimum cluster size** — only surface clusters with ≥ 10 related observations. Smaller clusters are noise.
4. **Naming** — cluster name = most-connected entity + "theme" (e.g., "Project Alpha + Customer Retention").
5. **Write** — one markdown file per cluster in `{vault}/00 - Clusters/`, overwriting the previous version.

**Clusters are regenerated weekly.** Don't trigger on every session.

## Stability guarantees

- Entity index is **rebuilt**, not mutated — never risk corruption. Build to `.new`, swap atomically.
- Cluster notes are **overwritten** weekly, but always include a `previous-version: <backup-path>` field so Brian can see what changed.
- Memory-refs in Obsidian are **immutable** — once written, never overwritten. If an observation is re-indexed with a new ID, we add a new note, not replace.

## When to run `build-entity-index.sh`

- **Manual**: Brian runs it via `/entity-rebuild` slash command (if you add one later)
- **Automatic**: weekly on a cron/launchd job (setup instructions in README)
- **On-demand**: after a heavy session where many new entities likely emerged
